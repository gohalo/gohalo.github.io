简介 #

DevOps 工具的单点登陆与用户管理系统，这里包含了两部分相关内容：

  • 用户权限管理，通过分组方式可以设置某个用户所属的用户组，正常只允许一个用户属于一个用户组，但是允许设置多个；而权限的分配则是针对用户组设置的。
  • 细粒度鉴权，也就是某个用户组所允许的操作，同时用户组不同角色的权限也不相同，而且不同用户组只能访问该用户组的资源。

通过上述的方式基本可以实现分权分域的策略，同时可以保证一定的灵活性。其实核心的部分主要是身份标识 (你是谁)、认证 (怎么证明)、授权 (能做什么)、审计 (做过什么)。

核心特性 #

-----> 用户管理
1. 注册(允许不同登录方式)、注销(可配置是否真正删除)、修改。
2. 获取当前用户详情。
3. 分页获取用户列表。
4. 账号锁定。
5. 用户组管理，由超级管理员进行分配。
6. 不同鉴权方式，例如人机、机机等。

-----> 权限管理
1.

-----> 公共功能
1. 操作审计。

-----> 高级特性(云平台)
1. 权限委托，某个用户需要操作某个资源，资源管理者临时对改用户进行授权；或者有三方平台管理其它用户资源。
2. 计费场景，不同服务对某个用户会收取不同的费用，而上层的 Pass 会依赖底层 Iass 的资源。

一个初始用户具有只读权限，此时不会赋值任何权限，不同平台决定如何对该用户进行处理。

基本概念 #

权限管理有几个基础的概念：

  • 主体 Subject，一般对应了真实登录的用户，在一些机机鉴权场景中也可能是某个服务。
  • 客体 Object，被访问的资源，也可能与操作有关，例如可以查看但是不能修改。
  • 规则 Policy，这就与具体策略有关了，可以是 ACL、RBAC 等等。

平台系统 #

其中比较关键的是用户和角色，为了方便进行管理可以对其进行分组，在管理和实现上，相比来说可以减少成本。

一个用户权限系统可以划分成四大模块：

  • 登陆系统，包括了单点登陆。
  • 用户管理，会按照行政部门或者业务线划分，相同部门会有相似的权限。
  • 角色管理，与业务相关而预先设定的角色，在业务上线预先设定，通常不会修改，有点类似用户的标签。
  • 权限管理，相比来说更加复杂，包含了目标、操作。

权限划分越细意味着管理成本越高。

登陆系统 #

涉及到登录用户名以及密码管理，用户名通常有工号、手机号、邮箱、自定义用户名等几种方式。

用户管理 #

基本功能

  • 管理员账号，该账号用于用来初始配置，权限最大，而且不会被删除，通常来说只有一个。
  • 账号禁用、停用，禁用意味着临时停用，通常因为多次输入密码错误、账号盗用、强制修改密码等；停用一般是因为员工离职。
  • 离职流程，一般会涉及到权限交接。

具体实现

  • 用户信息，密码加固策略。
  • 安全加固，防暴力破解、
  • 账号状态，通过整型表示不同原因，包括密码错误、盗用、强制修改密码等。

还有一个就是临时用户，最常见的是访客。

角色管理 #

通常一个系统中用 super、admin、reader、guest 基本已经足够，而且 guest 可以完全忽略；如果需要更精细化的管理，那么角色最好也通过层级进行管理。

另外，如果出现了层级，那么有些权限需要继承；而有些权限可能是互斥的。

权限管理 #

  • 自动赋权，在加入到某个部门后自动赋予该部门所具有的基础权限，实现时需要考虑后续会新增业务，该业务可能也会配置给某个部门。

其它 #

人机帐号和机机帐号实际上是两种不同的场景：

  • 人机帐号，自然人进行登录，登录需要鉴权，允许自动登出、帐号锁定等操作，后台需要维护 Session 信息，而用户的登录了信息可以通过 JWT、Cookies 等方式保存。
  • 机机帐号，一般通过 AppCode/Secret 或者 AK/SK 方式登录，没有登出、锁定的限制，但是需要定期轮转，可以通过 Token 实现。

那么这就涉及到了上述所说的两种登录信息维持方式了。

  • Session 常用于人机帐号，通过用户名密码登录，通过后台保存会话信息，依赖共享存储，可以立即登出。
  • Token 两者都可以，常用于机机帐号，不依赖其它组件，但是包含鉴权信息有限。

发展历史 #

从最早期的自主访问控制 (Discretionary Access Control, DAC) 和强制访问控制 (Mandatory Access Control, MAC)，再到中期基于角色的访问控制 (Role-Based Access Control, RBAC)，以及在一些复杂场景下使用的。

现在基本在讨论的是 Zero-Trust 以及细力度鉴权。

策略 #

XACML 是一个比较成熟的鉴权方案，只是其表达式看起来实在是有些繁琐，相比于 AWS IAM 的 Policy 来说要简单很多，但是一些常见的概念可以参考。

其它 #

机机 VS. 人机 #

分别对应了服务以及自然人的调用，两者有如下的区别：

  • 机机通过 AppCode 或者 AK/SK 进行访问，无需登录；人机需要登录，并在浏览器中保存会话标识信息。
  • 机机可以通过 Token 定期刷新即可，一般分钟级；人机允许立即登出，会话有效期为天。
  • 机机可以通过 API 调用任何接口，当然也可以进行权限控制；人机则会受权限控制。

实现方式 #

基于JWT #

通过 JWT 实现的一种基于 Token 鉴权方式，在 Token 中包含了角色信息，不同 API 会检查不同的角色，其中 JWT 负载部分包含了如下内容。

{
  exp: 1609654808,
  role: ["admin", "readonly"],
  uid: "g001003"
}

实现介绍 #

内部包含了角色和用户两个表，通过 type Backend interface 接口定义 Backend 实现，包括了授权、获取用户信息等接口。

这里仅提供了一个通过内存保存用户信息的实现，对应了 login/simple.go 中的实现，包含了两个表 (或者说是缓存)：A) roles 角色信息，例如 admin readonly 等；B) users 用户信息，例如 UID、用户名、密码等信息。

内置的角色和用户详见如下。

密码验证 #

保存用户密码时，必须要保证安全，简单来说，就是所谓 One-Way Hash 算法，这里采用的是 golang.org/x/crypto/bcrypt 中的实现，不过其计算比较耗时。

最佳实践 #

设计时通常会包含两个特殊的角色，可能名字不同：

  • admin 只具备基础的只读操作，但可以管理用户权限等。
  • super 超级用户，除了不能管理用户之外，可以执行任何操作，所以，密码需要妥善管理。
