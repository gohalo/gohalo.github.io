审计日志 #

审计日志，包括了登录(login-Done)、执行命令(shell-Done)、请求信息(request)，另外对于交互会话的执行也会记录，对于端口转发、X11等的保存成本有点高。

登录信息 #

配置主机登录信息有两种方式：A) 如果已经绑定主机组，那么以主机组的配置为准；B) 未绑定主机组，可以绑定登录信息。

密码更新 #

前提条件：A) 支持两种管理帐号登录，用于防止其中一个更新失败进行恢复；B)

更新时按照主机组挨个登录方式进行。

已知问题：A) 修改过程中如果该用户只有这一种登录方式，那么可能会登录失败。例如，admin 用户只配置了密码登录，那么因为更新过程中不确定是否更新完成，就可能出现报错。

建议使用公私钥或者证书方式登录而不是密码方式，因为如果涉及到更新那么密码会被修改，可能会影响其它系统使用该用户。

重要 #

  • 支持用户登录时密钥的重新绑定，包括 HTTP(TODO)、Shell 实现。
  • 用户登录时密钥支持过期以及状态配置。
  • HTTP 和命令行参数校验统一。
  • 基于角色和策略的用户管理、鉴权方式如何处理，能否得到兼容，或者根据用户策略可配置。
  • 双因子认证，目前只能在登录之后验证。
  • 对于签名不一致的主机，提示用户允许更新，详见 checkHostKey() 的实现。
  • 通过Shell登录时支持调试参数，用来排查可能的异常。
  • 支持 IPv6 协议，包括服务端以及登录远端服务器。
  • 使用对称加密的时候，根密钥文件当前使用的是明文，需要进行加密处理。

其它 #

  • 数据库写入支持 Create/Update 更新，当前直接忽略，但是在 checkHostKey 中无法通过 Omit 忽略。
  • 编译脚本通过 Makefile 重构，便于打包测试等。
  • Bootstrap 命令添加检查参数，确保初始化是正常的。
  • database 子命令实现，需要同时考虑版本判断，并通过用户确认。
  • 链接到服务端时可以选择会话复用，好处是连接数减少，但是如果某个链接异常可能会导致其它异常。
  • 日志 Trace 统一，当前通过函数 String 参数传递有点难看，建议通过 context.Context 实现。
  • HTTP 审计日志采用 zap 日志保存，而不需要 sugar 实现。
  • Shell优化，参考 Grumble 的实现，同时也包含了readline的实现。
