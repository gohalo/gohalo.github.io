开发规范 #

1. 统一错误码 #

错误码采用 32bit 的无符号整型，也就是最大为 4294967295 ，因为错误码大部分时间需要用来排查问题，如果采用十六进制表示，对于用户来说不友好，所以采用十进制标示，总共 10 位，区分如下。

429 496 7295
000 000 0000

高三位 分类 BootAgent 003 BasicAgent 006 MonitorAgent 009
中三位 由不同的分类进行重新划分
低四位 具体的错误码信息，可以用有限的16bit标识

具体的错误信息，可以参考对应的开发库。另外，对于 REST API 来说，返回的错误信息格式为。

{
    "code": 0,                         统一错误码
    "message": "invalid parameters",   统一错误信息，是错误码的解释
    "cause": "failed to login",        具体的错误信息
}

2. Issues 管理 #

也就是 issues 中使用的标签，主要分为如下几类。

  • Feature 新特性，用于讨论想要添加功能。
  • Bug 问题描述以及解决方案讨论。
  • Security 安全问题，但是不在公开平台讨论，仅作为隐藏标签。

3. 版本发布 #

这里采用的是语义化的版本控制，每次发布使用 tag 标记，详细可以参考 semver.org 中的介绍，例如 BootAgent-1.2.3-4.x86_64 ，主要包含了：

  • 主版本号，做了不兼容的API修改，一般是增加了一些重大的特性修改。
  • 次版本号，向下兼容的功能性新增，也可以是针对一些特性的优化，但是不影响原有的接口。
  • 修订号，向下兼容的问题修正，一般是 bugfix 。
  • 打包号，在集成验证时打包的版本次数。

上述的内容为正式的版本发布，在迭代发布的过程中，还包括了 alpha beta gamma/stable 三个开发阶段，一般定义为。

  • alpha 处于开发测试阶段，常用于各个组件之间的联调测试，注意，可能存在 bug，功能会被删除，接口会被修改，不会另行通知。
  • beta 各个团队已经基本完成开发，可以在测试环境进行功能联调，接口不会删除，细节会有调整以及 bugfix 等。
  • gamma 基本可以发布了，可以在预发环境进行验证了，对应的功能会在后续版本中发布。
  • yanked 已经发行版本，但是因为风险被取消。

其中的 yanked 在版本发布之后用来取消。

发布的内容一般包含了如下的内容。

/BootAgent/BootAgent-1.2.3.tar.bz2                   源码包
/BootAgent/BootAgent-1.2.3-1.x86_64.rpm              RPM安装包
/BootAgent/BootAgent-debuginfo-1.2.3-1.x86_64.rpm    调试信息

4. 测试规范 #

4.1 宏定义 #

有几类的默认宏定义：

  • __DEBUG__ 只有在调试模式使用；
  • __TEST__ 测试时使用，当前只有单元测试；
  • __MOCK__ 考虑到运行时的性能，通过宏的方式定制。

4.2 STATIC_TESTABLE #

很多的一些函数定义的是内部可见，也就是通过 static 声明，为了方便进行测试可以将该函数导出，定义如下。

#ifdef __TEST__
#define STATIC_TESTABLE
#else
#define STATIC_TESTABLE static
#endif

4.3 通用测试 #

  • 时间相关，夏令时、冬令时、闰年、闰秒、时钟手动修改。

发布规范 #

1. 命名规范 #

1.1 主机命名 #

希望可以通过主机名称可以快速识别相关信息，包括所属机房信息、业务信息等，如果是多云托管，那么最好也包含，所以，主机命名规则为 云服务提供商.Region代号AZ代号.业务域.服务名.组件名.主机编号，大小写敏感，建议使用驼峰命名。

注意，各个组件命名中不能有 .，建议使用驼峰命名，也可以通过 - 或者 _ 分割，例如 AWS.CNNorth1A.SRE.CMDB.Console.001 。

1.2 变量命名 #

这里的命名规范包括了 Tags 的命名，也被称为变量名，可用字符为 ASCII ，包括了 A-Z a-z 0-9 _ @ # 。

2. 系统用户 #

系统用户的终端一般设置为 /bin/false 或者 /sbin/nologin ，在进行测试时，如果只执行 su - monitor -c "xxx" 一般不会有输出，应该通过如下方式执行。

# su - monitor -s /bin/bash "pwd"
