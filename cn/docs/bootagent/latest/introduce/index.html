简介 #

主要目的是用来管理最基础的 Agent (例如 BasicAgent、MonitorAgent 等等)，提供基本的管理、监控等功能。简单来说，尽量保证 BootAgent 功能的简单稳定，这也同时意味着，很多的功能无法保证其时效性。

通常在服务器的安装镜像中已经预置的 BootAgent 程序，启动后会自动连接到服务器进行注册。

编译时需要安装 yum install openssl-devel libcgroup-devel libuuid-devel 开发库。

最小依赖 #

为了保证绝大部分场景下 BootAgent 都可以正常运行，会简化大部分的场景，以尽量降低依赖。

  • 不支持域名解析，仅支持 IP 地址直接访问，包括了连接初始化、下载安装包等。

基本功能 #

工作时有几种模式，这里介绍的是正常模式下的基本功能。

1. BootAgent 包括配置、自身监控等
   1.1 修改配置，这里只包含了有限配置信息，包括了 AgentSN、Tags、ServerList 。
   1.2 任务信息统计，接收到的任务数、执行成功数、执行失败数、忽略执行数(任务已经存在)。
   1.3 进程状态，包括 CPU、内存、句柄、磁盘 IO、网络。

2. 子进程管理。
   2.1 任务接口，包括了子Agent的安装、卸载、启动、停止。
   2.2 状态管理。
       自动拉起，可以配置是否在启动时拉起进程。
       状态检查、资源限制。
   2.3 事件上报，例如重启。

如上所述，在实现时尽量做到简化，所以必然会带来很多的限制，简单列举如下：

  • 配置保存，只通过简单的文件保存，原子操作基于文件系统的移动。
  • 地址信息，不使用域名而是只有 IP 地址，这样不会依赖 DNS 服务。

另外，在实现时，同样尽量保持简单：

  1. 与服务端通讯的发送、接收缓存默认是 64K 大小，通过 CLI_SNDBUF_SIZE CLI_RCVBUF_SIZE 设置。
  2. 客户端会周期将状态上报到服务端，默认 10min (可配 1min~60min )，通过 PROJECT_CLIENT_INTERVAL 设置。
  3. 如果在一个周期内数据上报失败，那么会减少到 3min (可配 1min~60min) 重试上报，可通过 PROJECT_RETRY_DELAY 设置。

注意，客户端上报周期并非严格设置，可能会在连接服务端失败时导致延迟。

安装部署 #

----- 配置文件
/etc/cargo/BootAgent/BootAgent.conf
/etc/cargo/BootAgent/state/agents      # 保存Agent相关的期望状态
/etc/cargo/BootAgent/state/mode        # BootAgent的运行模式
/etc/cargo/BootAgent/items/AgentSN     # 可通过服务端主动配置
/etc/cargo/BootAgent/items/Tags
/etc/cargo/BootAgent/items/ServerList

----- 二进制程序
/usr/sbin/BootAgent
/usr/sbin/BootAgentCtrl

----- 基目录以及临时目录，常见如保存下载文件
/usr/local/cargo/BootAgent
/usr/local/cargo/BootAgent/tmp         # 临时目录，会有任务定期清理
/usr/local/cargo/BootAgent/tmp/MiniAgent-1.2.1.tar.gz

----- 日志以及PID文件
/var/log/cargo/BootAgent.log
/var/run/cargo/BootAgent.pid
/var/run/cargo/BootAgent.sock

----- 文档以及特定的配置文件等
/usr/share/cargo
/usr/share/doc/cargo/BootAgent/AUTHORS
/usr/share/doc/cargo/BootAgent/COPYING
/usr/share/doc/cargo/BootAgent/ChangeLog
/usr/share/doc/cargo/BootAgent/README
/usr/share/doc/cargo/BootAgent/TODO
/usr/share/man/man1/BootAgent.1.xz
/usr/share/man/man5/BootAgentCtrl.5.xz
